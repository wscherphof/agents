Running cloud coding agents against repos the cloud can't reach

A tiny launcher repo that lets Claude Code Web sessions work on Azure DevOps projects — and any other repo Anthropic can't see directly.

View the repo on GitHub →

Claude Code Web is great when it works on your repo. The catch: Anthropic ships a GitHub App that lets cloud sessions reach GitHub repos, but there's no equivalent for Azure DevOps. If your code lives on ADO, you're locked out of the browser-based agent experience.

The agents repo is a small, clever workaround. It's not a project codebase — it's a launcher you start a Claude Code Web session from. It lives on GitHub (where Web sessions work), and on startup it pulls your real project repo into reach. The result: full cloud agent sessions against repos Anthropic can't otherwise touch.

The core trick

When a session boots on a branch of the launcher, a session-start hook does the heavy lifting:

  • It reads which project to target from variables at the top of the hook.
  • It clones (or fast-forward pulls) that project repo into a git-ignored src/ folder, authenticating with a Personal Access Token from the session environment.
  • It runs per-project setup steps — installing dependencies, pinning versions, and on Azure DevOps, installing the az CLI in the background so the agent can open PRs.
  • It mirrors the project's agent configuration back into the launcher.

That last step solves a subtle problem. Claude Web only reads agent config — CLAUDE.md, .claude/, MCP servers, custom agents — from the launcher repo it boots into, not from the cloned project. So the hook syncs the project's config into the launcher, then commits and pushes it to a stable per-project settings branch. The next session for that project picks it all up automatically, and the configuration accumulates and improves over time.

The big idea: the launcher lives where cloud sessions can run, and reaches out to where your code actually is.

A branch per project

You pick which project a session drives simply by choosing a branch in the Claude Code Web branch picker. The branches form a tidy hierarchy: 

main
├── geowep              # whole GeoWEP repo
│   └── geowep/ng       # GeoWEP "ng" component
└── <next-project>
    └── <next-project>/<component>
  • main — the base scaffolding template, tied to no project.
  • Project branch (e.g. geowep) — configured for a whole repo, carrying its mirrored root settings.
  • Component branch (e.g. geowep/ng) — configured for one component of a monorepo, with that component's settings layered on top.

Nothing to install

There's no local setup. You fork the repo on GitHub and run everything from your fork. Project variables, mirrored settings, and PATs all live there — nothing touches the upstream repo. To pick up improvements later, you sync your fork's main and merge it down into your project branches, so every project gets launcher updates without losing its own config.

The one piece of configuration you do supply is a Personal Access Token, set in the cloud Environment as GITHUB_PERSONAL_ACCESS_TOKEN or AZURE_DEVOPS_EXT_PAT. Since the token scopes access to a project, the usual pattern is one Environment per project, selected alongside the project's branch when you start a session.

Two repos, two git workflows

A session juggles two repos at once, and the trick to staying sane is keeping their git workflows separate:

The launcher The project repo (src/)
Holds Agent scaffolding + mirrored settings The actual code you're working on
Edits land on The settings branch — pushed automatically by the hook A new session branch, or an existing feature branch
PRs n/a — settings auto-push Opened from the session branch, or pushed onto an existing one

The agent handles the project-repo branching and PR itself. As a developer, your starting prompt decides the route: describe a new bug or feature and it spins up a fresh branch; ask it to continue an existing feature branch and it checks that out and pushes straight onto it. Either way, you just review and merge the resulting PR.

One thing worth remembering: cloud sessions run in ephemeral containers, so only pushed commits survive. The agent is instructed to always push after committing — but it's a useful detail to keep in mind for how the whole system stays durable across one-shot sessions.

Why it matters

It's a small repo with an outsized payoff. By treating a GitHub-hosted launcher as a bridge, it brings the full Claude Code Web experience — cloud sessions, automated PRs, accumulating per-project agent config — to codebases that the browser agent couldn't otherwise reach. If your team's code lives on Azure DevOps and you've been watching cloud coding agents from the sidelines, this is the on-ramp.
